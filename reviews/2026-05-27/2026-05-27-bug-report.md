## 批改报告

### 逐项评价

| 要求 | 完成度 | 点评 |
|------|--------|------|
| 1. Bug 标题 | 60% | 描述了问题现象但缺少位置信息。标准格式："【模块/页面】条件 + 现象"，如"【百度搜索】输入框为空时点击搜索按钮，无任何提示"。 |
| 2. 复现步骤 | 75% | 步骤基本清晰，从打开浏览器到触发 Bug 路径完整。Step 1-2 可合并为"用 Edge 浏览器打开 https://www.baidu.com"。 |
| 3. 预期 vs 实际 | 90% | 两者都明确写出，格式清晰。"预期结果"具体（"输入不能为空"），"实际结果"也准确描述了现象。 |
| 4. 严重级别 | 50% | "低"判定偏低——功能提示缺失属于功能性缺陷而非纯体验问题，应判为"**一般**"。理由"体验感不佳"太笼统，应说明"不影响搜索主流程但缺少输入校验反馈"。 |
| 5. 环境与附件 | 40% | 提供了操作系统/浏览器/版本号，但缺少截图（题目明确要求"附上截图或环境信息"）。 |

### 总分：63/100

### 具体问题

1. **Bug 标题过于简略**：只写了"输入为空，给予提示"，没有说明这是哪个系统/哪个页面的问题。标准 Bug 标题应包含"在哪"+"什么操作"+"什么现象"，如"【百度搜索首页】搜索输入框为空时点击'百度一下'按钮，页面仅刷新无校验提示"。
2. **严重级别评定不准确**：判为"低"（次要）不合理。输入校验提示缺失属于功能性缺陷（程序未做其应该做的），应判为"一般"级别。真正"低/次要"的问题如页面字体偏小、按钮颜色不一致——不影响功能使用，纯体验问题。
3. **缺少截图**：题目明确要求提供截图或环境信息。只有文字环境信息不够，Bug 报告中截图是让开发快速理解问题的关键。

### 修正参考

```
Bug 标题：【百度首页】搜索框为空时点击"百度一下"按钮，无输入校验提示

严重级别：一般（Minor）
判定理由：功能提示缺失，用户不知道为何搜索无反应，但刷新后可继续使用，不阻塞主流程。

复现步骤：
1. 使用 Edge 浏览器打开 https://www.baidu.com
2. 不输入任何内容，直接点击"百度一下"按钮

预期结果：页面提示"请输入搜索内容"或类似校验文案
实际结果：页面仅刷新，无任何提示

环境信息：
- 操作系统：Windows 11 Home China 10.0.26200
- 浏览器：Microsoft Edge 140.0.3967.83 (64 位)
- 网络：正常联网
- 截图：见附件
```

### 改进建议

- 标题写"在哪 + 什么条件 + 什么现象"三段式，5W 模板：谁(who)在什么条件下(when/where)做什么(what)出现什么(how)
- 严重级别判断看"阻塞程度"而非主观感觉：崩溃=测不下去，严重=有变通方案但核心功能受影响，一般=功能可用但有小毛病，次要=纯体验/建议
- 提交 Bug 时截图必备——至少截一张触发 Bug 后的页面全貌
